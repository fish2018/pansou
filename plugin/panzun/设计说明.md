# 盘尊社区插件设计说明

## 插件位置

- 源码：`plugin/panzun/panzun.go`
- 站点：`https://www.panzun.cc`
- 插件名：`panzun`

## 目标

将 **盘尊社区** 接入为 PanSou 搜索插件，当前目标聚焦于：

1. 根据关键词搜索盘尊社区帖子
2. 进入帖子详情页提取网盘链接
3. 将 `a.7u9.cn` 短链尽可能解析成真实网盘分享地址
4. 向 PanSou 返回 **真实网盘地址**，而不是中间短链

当前版本**不追求完整论坛功能**，只实现 PanSou 需要的“搜索 → 返回真实链接”主链路。

---

## 站点结构判断

盘尊社区并不是普通 HTML 资源站，而是一个 **Flarum API 论坛**。

### 关键 API

- 搜索主题：
  - `GET /api/discussions?filter[q]=关键词&page[offset]=0`
- 详情主题：
  - `GET /api/discussions/{discussionId}`

### 主题数据来源

插件不依赖前端页面 DOM，而直接使用 API：

- 搜索接口返回 discussion 列表
- 详情接口返回：
  - 主题标题
  - 附带 `included` 资源
  - 首帖 `contentHtml`

插件核心就是从 `contentHtml` 中提取真实网盘链接。

---

## 当前实现方案

## 1. 搜索流程

`searchImpl()` 的逻辑：

1. 用关键词请求 `/api/discussions?filter[q]=...`
2. 最多拉取前几页（当前代码里限制了最大页数）
3. 遍历 discussions
4. 对每条 discussion 调详情接口
5. 从详情页首帖正文中提取网盘链接
6. 只保留 **有真实链接** 的结果
7. 转为 PanSou 标准 `model.SearchResult`

### 返回字段

主要返回：

- `UniqueID`
- `Title`
- `Content`
- `Datetime`
- `Tags`
- `Links`

其中 `Links` 是最关键部分。

---

## 2. 详情提链逻辑

详情接口返回的 `included` 中含有 `posts` 资源。

当前逻辑：

1. 找首帖 post
2. 取 `contentHtml`
3. 在 HTML 中做正则提取
4. 提取两类链接：
   - 真实网盘链接
   - `a.7u9.cn/s/...` 短链

### 当前识别的网盘类型

- 夸克：`pan.quark.cn`
- UC：`drive.uc.cn`
- 百度：`pan.baidu.com`
- 天翼：`cloud.189.cn`
- 阿里：`aliyundrive.com` / `alipan.com`
- 迅雷：`pan.xunlei.com`
- 115：`115.com`
- 123：`123684.com` / `123865.com` / `123912.com` / `123pan.com`

---

## 3. 短链解析逻辑

盘尊社区当前的关键难点是：

### `a.7u9.cn` 并不总是直接 302

实测发现该短链服务会根据请求头、环境、运行上下文表现出不同响应：

#### 情况 A：理想情况
直接返回：

- `302 Found`
- `Location: https://pan.quark.cn/s/...`

这是最容易处理的情况。

#### 情况 B：中间页情况
返回：

- `200 OK`
- 一个带 JS 的 HTML 页面
- 页面中包含：
  - `_s = 'xxxx'`
  - `_v = 'xxxx'`
  - 以及疑似接口路径 `api_v1_go_collect/info`

这说明它并不是纯后端 302 服务，而有一层 JS / 验证逻辑。

### 当前插件内的处理策略

`resolveShortLink()` 当前做的是**多策略尝试**：

1. 尝试用简单请求头获取 302 `Location`
2. 尝试用 cloudscraper
3. 尝试普通 no-redirect client
4. 如果拿不到真实地址，则返回空，后续丢弃该结果

### 当前结论

盘尊短链解析**最关键的成功条件**是：

> 使用尽量简单的请求头，避免触发中间页

例如：

- `User-Agent: python-requests/2.31.0`
- `Accept: text/html,*/*`
- `Accept-Encoding: identity`

在某些环境下，这样能直接拿到 302 的真实 `Location`。

### 当前仍存在的风险

如果服务器始终返回 JS 中间页，而不再直接给 302，那么插件当前实现可能无法稳定解析短链。

也就是说，**盘尊插件的最主要不确定性就是 7u9 短链解析链路**。

---

## 4. 为什么插件不直接返回短链

PanSou 目标是聚合真实网盘资源。

如果直接返回 `a.7u9.cn/s/...`：

- 下游系统无法识别网盘类型
- 无法统一分类为 quark / baidu / aliyun 等
- 用户看到的是中间短链，而不是真实分享地址

所以插件设计目标明确为：

> **只返回真实网盘地址，不返回 7u9 短链本身**

这也是当前实现里，若短链解析失败，则该链接不会作为有效结果输出的原因。

---

## 5. 插件优先级与过滤策略

当前插件采用：

- `plugin.NewBaseAsyncPlugin("panzun", 2)`

原因：

- 盘尊社区内容质量整体较高
- 资源标题相对规范
- 网盘分享较丰富
- 但短链解析存在一定不稳定性，因此暂未给到最高优先级

过滤策略目前仍保留 Service 层关键词过滤。

如果后续发现：

- 标题宽泛
- 一条帖子中含多个作品
- 或搜索结果经常因二次过滤误删

可考虑改成：

- `NewBaseAsyncPluginWithFilter(..., true)`

即跳过 Service 层过滤。

---

## 6. 当前实现的局限

### 1）依赖详情页逐条展开

搜索接口本身不直接返回真实网盘链接，因此必须：

- 搜索 discussion
- 再请求 detail
- 再从正文提链

这意味着请求成本高于纯 API 型插件。

### 2）短链服务行为不稳定

`a.7u9.cn` 可能：

- 直接 302
- 返回 JS 中间页
- 根据请求头不同表现不同

这会导致不同环境下解析结果不一致。

### 3）目前只做“搜索用途”

当前插件只为 PanSou 搜索设计，没有做：

- 首页
- 分类浏览
- 详情展示增强
- 播放/网盘目录展开

这些不是 PanSou 当前目标所必需的。

---

## 7. 后续优化建议

## 优先级最高

### A. 固化短链解析方案

如果后续确认某种请求方式最稳定（例如：
- 某组固定 headers
- 某种 client
- 某个 Cookie/Session 组合
）

应将 `resolveShortLink()` 收敛成最小稳定实现，减少试探式逻辑。

### B. 解析 JS 中间页的真实接口

如果 `a.7u9.cn` 最终不再给直接 302，应继续分析中间页：

- `api_v1_go_collect/info` 的真实路径
- 参数签名
- cookie 与 `_s / _v` 的关系
- 是否需要 POST / 特定 header

一旦该链打通，短链解析稳定性会显著提升。

## 次优先级

### C. 增加详情缓存

当前若同一个 discussion 重复命中，建议缓存：

- detail JSON
- 短链解析结果
- 提链结果

以减少重复请求。

### D. 增加帖子内多作品标题识别

若未来盘尊社区出现“一贴多作品”场景，可考虑为每个链接设置：

- `WorkTitle`

提高 PanSou 前端展示区分度。

### E. 增加更完整的时间字段处理

当前时间主要来自 discussion 的 `createdAt`。

可后续增强：

- 首帖时间
- 最后更新时间
- 网盘链接发布时间

---

## 8. 适用范围

这个插件适合：

- 搜索影视、动漫、综艺、音乐等网盘资源
- 返回真实网盘分享链接
- 作为 PanSou 中的一个论坛类数据源

不适合：

- 需要强实时解析复杂短链的场景
- 需要站内完整浏览体验的场景
- 依赖播放、目录展开的场景（这些属于 OmniBox 网盘源范畴，而非 PanSou 搜索插件）

---

## 9. 当前结论

`panzun` 插件的设计核心可以概括为：

> **用 Flarum API 搜帖子，用详情正文提链接，用短链解析拿真实网盘地址，最后只把真实链接交给 PanSou。**

当前最大技术难点不是搜索本身，而是：

> **`a.7u9.cn` 短链解析稳定性**

其余搜索、详情、提链主链路都相对明确。
